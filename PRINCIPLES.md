# Project Principles

This repository follows a principles-first model for quality, trust, and long-term maintainability.

## 1. Clarity over hype

Entries and docs must be factual, specific, and neutral.
Prefer concrete language over marketing language.

## 2. Structured truth

Critical metadata belongs in schema-validated YAML, not only in prose.
If something matters for comparison, it must be represented as structured data.

## 3. Evidence-driven curation

Classification and claims should be based on observable project behavior, docs, and repository artifacts.
When uncertain, state uncertainty explicitly.

## 4. Explicit limitations

Every entry should describe limitations, not only strengths.
Honest limitations improve decision quality.

## 5. Reproducibility

Indexes must be generated from source data through scripts.
Quality checks should be repeatable locally and in CI.

## 6. Accessibility for newcomers

Beginner guidance is a first-class requirement.
Documentation should support first-time contributors without lowering quality standards.

## 7. Maintainability at scale

Taxonomy, templates, and review standards should reduce entropy as the catalog grows.
Consistency is a feature, not overhead.

## 8. Security and responsibility

Community safety, responsible disclosure, and secure automation practices are non-negotiable.

## 9. Open collaboration

Contributions are welcome when they align with schema quality, taxonomy consistency, and community norms.
Feedback should be direct, respectful, and actionable.

## 10. Stewardship mindset

This project should remain useful over time.
Favor decisions that improve long-term trust, navigability, and contributor health.
